# RAVO v0.2 Implementation Review Proposal

Date: 2026-07-07

## Review Objective

Review whether the current RAVO v0.2 implementation is sufficient to satisfy `docs/ravo-v0.2-decision-complete-spec.md` without becoming a monolithic scheduler or overstating readiness.

## Product Scope

RAVO v0.2 is a modular Codex plugin suite. Stable plugin ids:

- `ravo-core`
- `ravo-analysis`
- `ravo-workstream`
- `ravo-quick-validation`
- `ravo-acceptance`
- `ravo-knowledge`

Modules must remain independently installable and connect only through `knowledge/.ravo/manifest.json` and artifact files.

## Implemented Changes

- Added `ravo-workstream` for milestone, status, blocker, recovery, decision, next-step, and evidence artifacts.
- Added `ravo-quick-validation` for smoke/phase evidence artifacts and smoke readiness checks.
- Added `ravo-knowledge` for workspace-local knowledge write/retrieve/apply and opt-in user-level transferable lessons.
- Upgraded existing plugin versions to `0.2.0`.
- Expanded `ravo-core` initialization to create `analysis`, `workstream`, `quick-validation`, `acceptance`, `continuation`, and `knowledge` directories.
- Added Goal Prompt Authoring support via `ravo-goal-prompt.js`, which emits a concise Goal prompt only after finding a decision-complete spec.
- Added `ravo-analysis/scripts/write-decision-spec.js` so decision-complete spec generation is mechanically available before Goal Prompt authoring.
- Expanded `ravo-acceptance` to discover workstream and quick-validation artifacts before reporting readiness.
- Preserved `UserPromptSubmit` and `Stop` as fallback/advisory mechanisms, not hard before-final gates.
- Updated README/README_ZH for six-module installation, hook trust, Codex global `AGENTS.md` integration, upgrade flow, and module boundaries.
- Added schemas for workstream, smoke, and knowledge artifacts.
- Expanded validation scripts and prompt regressions.

## Key Design Decisions

- No central dispatcher or scheduler.
- Workstream does not judge release readiness; it hands evidence to acceptance.
- Smoke evidence is not final acceptance.
- Raw workspace facts and evidence stay workspace-local by default.
- User-level transferable lessons require explicit opt-in, redaction checks, applicability, confidence, and canary leakage rejection.
- Goal Prompt should reference a complete spec instead of duplicating the spec.
- RAVO does not silently modify Codex global `AGENTS.md`; installing Agents must show a diff and ask for approval.

## Known Risks

- Hook trust is host-dependent. Without trusted hooks, natural governance degrades to weaker manual or skill-triggered behavior.
- Real Codex session/subagent E2E validation is required because scripts cannot fully prove proactive agent-initiated acceptance.
- External model review previously had provider HTTP 502 failures during spec review; current implementation review may still be partial if the provider is unavailable.
- The implementation uses small Node stdlib scripts and JSON artifacts; this is intentional, but future versions may need stronger schema validation if adoption grows.

## Review Questions

1. Does the implementation satisfy the v0.2 required module contracts without exceeding the non-goals?
2. Are any required validation paths still insufficient or too script-only?
3. Is the acceptance/readiness boundary coherent, especially around `pending_acceptance`, smoke evidence, and real E2E?
4. Is the knowledge transfer model safe enough for v0.2, given the opt-in and redaction/canary checks?
5. Are README and AGENTS integration instructions clear enough to avoid users modifying project-level `AGENTS.md` by mistake?
