# RAVO v0.1 设计决策

日期：2026-07-06

## 结论

v0.1 只实现 `ravo-core`、`ravo-analysis`、`ravo-acceptance`，并通过 `knowledge/.ravo/manifest.json` 和 JSON artifacts 连接模块。暂不实现完整 `workstream` 和 `knowledge` 模块。

RAVO 的交互目标是自然触发：用户正常描述需求、问题、根因或验收诉求时，系统应通过语义自动启用合适能力；显式说出 skill/plugin 名称只应作为测试和调试手段，而不是日常使用要求。

验收的主触发必须是 Agent 在交付/发布结论前主动发起，而不是等待用户 prompt 询问。v0.1 用 `SessionStart`/`SubagentStart` 注入主动验收规则，用 `UserPromptSubmit` 作为用户直接询问验收/发版/上线时的兜底门禁；当前未找到可拦截 Agent 最终回复的 Codex hook。

## 第一性原理分析

- goal：用最小可安装套件验证 RAVO 的开源产品形态。
- constraints：插件必须可独立安装；不能默认修改 `AGENTS.md`；不能做中心化调度器；小任务不能被全流程拖慢。
- facts：Codex 插件可作为多个 marketplace entry 发布；skills/hooks/scripts/templates 可随插件分发；跨插件最稳定的连接方式是 workspace artifact。
- root cause：RAVO 的主要产品风险不是缺能力，而是耦合过重、安装摩擦过大、状态证据不可信，以及要求用户刻意调用插件导致交互不自然。
- derived conclusion：先发布统一品牌下的模块化插件，通过 artifact discovery 自动增强；能力触发应优先依赖 skill description 和轻量语义 hook，而不是要求用户手动 call。

## 已实现

- repo-local marketplace：`.agents/plugins/marketplace.json`
- 三个独立 plugin entry：`ravo-core`、`ravo-analysis`、`ravo-acceptance`
- 共享 schema 和模板
- `knowledge/.ravo/manifest.json` 初始化
- analysis artifact 写入
- acceptance artifact 写入和检查
- agent-initiated acceptance skill contract
- session/subagent proactive acceptance context
- acceptance prompt gate fallback
- natural-language prompt semantics as the intended default trigger path
- opt-in `AGENTS.md` diff preview / backup / idempotent apply
- atomic writes for shared JSON artifacts
- `AGENTS.md` restore from backup
- smoke test

## 故意不做

- 不实现 `ravo-workstream` 完整模块。
- 不实现 `ravo-knowledge` 完整模块。
- 不做中心化 dispatcher 或 state bus。
- 不把长模板塞进 `AGENTS.md`。
- 不静默修改任何用户规则文件。
- 不把显式调用 RAVO 作为正常用户路径。

## 后续触发条件

- 当 v0.1 有真实使用反馈后，再做 `ravo-workstream`。
- 当 artifact 数量开始影响检索和复用时，再做 `ravo-knowledge`。
- 当真实 prompt 测试发现 skill description 自动触发率不足时，优先补 `ravo-analysis` advisory hook，而不是要求用户显式 call。
- 当 schema 需要破坏性变更时，再进入 `0.2` 或 `1.0` 版本策略讨论。
