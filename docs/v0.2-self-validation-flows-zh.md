# RAVO v0.2 自验收多轮对话用例

这份文档用于你自己验证 RAVO v0.2 的真实交互效果。

核心原则：**不要像考试一样问“请测试 RAVO 是否触发”。**  
用一个真实的小需求，从需求判断、规格、执行、验收、复盘一路走完，看 Codex 是否自然表现出 RAVO 应该带来的行为。

## 准备

建议新开一个 Codex session，并在一个临时空目录里测试，例如：

```bash
mkdir -p /tmp/ravo-v02-user-check
cd /tmp/ravo-v02-user-check
```

如果刚安装或升级过 RAVO，可以先问一次安装状态；这一步可以显式提 RAVO，因为它是安装检查，不是业务测试：

```text
看一下当前 Codex 环境里 RAVO 这几个插件是不是都已经装好了；如果要新开会话或者授权 hooks，也直接告诉我。
```

预期：

- 能看到 `ravo-core`、`ravo-analysis`、`ravo-workstream`、`ravo-quick-validation`、`ravo-acceptance`、`ravo-knowledge`。
- 会提醒新开 session。
- 会提醒 hook trust/授权会影响自然触发。
- 不会静默修改全局 `AGENTS.md`。

下面开始正式业务流。后续 Prompt 不要再写 `RAVO`、`plugin`、`skill`。

## 一条真实小需求闭环

背景：你想做一个很小的本地脚本，帮自己在小项目结束时生成交付说明。

它足够小，Codex 可以真的实现；也足够完整，可以覆盖需求分析、规格、长任务、验证、验收、复盘和知识沉淀。

## 第 1 轮：先判断需求，不要急着写代码

```text
我经常在做小项目时，最后不知道怎么整理交付说明。想做一个很小的本地脚本：我把这次改动和验证过程写到 changes.txt，它帮我生成一份 release-note.md。先别写代码，你先判断这个需求有没有必要做，真正用的人是谁，应该做到什么程度。
```

预期：

- Codex 不应直接开始写代码。
- 应识别真实消费者：写交付说明的人，以及读交付说明的人。
- 应区分事实、推断、假设。
- 应给出至少两个方案，例如“纯模板脚本”“更智能的解析”“先用手写模板”。
- 应挑战首选方案，例如指出这个脚本可能不如一个模板文件划算。

失败信号：

- 直接创建文件。
- 只顺着用户说“应该做”，没有挑战必要性。
- 没有说明边界。

## 第 2 轮：收边界，并判断是否需要规格书

```text
我感觉这个东西容易做大，你帮我收一下边界。哪些先不做？如果后面要让 Codex 自动开发，你觉得现在信息够不够？不够的话先整理一份规格，再给我一个短的 Goal Prompt。
```

预期：

- 会明确非目标，例如不做 Web UI、不接 AI、不做 git diff 解析、不做发布平台集成。
- 如果信息不够，应先补规格，而不是硬写超长 Goal Prompt。
- Goal Prompt 应短，主体引用规格文档或规格内容，不把所有需求重复塞进去。

失败信号：

- Goal Prompt 变成很长的 PRD。
- 没有输入/输出、验证方式、失败处理。
- 继续扩大范围。

## 第 3 轮：开始最小实现

```text
可以，按最小版本做。用 Node 就行，不要引入依赖。输入文件就是 changes.txt，输出 release-note.md。先把任务拆一下，说明第一步做什么，然后直接开始。
```

预期：

- 会拆出很短的执行步骤。
- 应留下下一步、当前阶段或类似 workstream 的跟踪信息。
- 实现应很小，不引入依赖。
- 不应做配置系统、复杂模板引擎、CLI 框架。

失败信号：

- 引入依赖。
- 搭了过大的项目结构。
- 只有口头计划，没有实际下一步。

## 第 4 轮：让 Codex 自己验证

```text
继续做，写完后自己跑一下。不要让我来测。
```

预期：

- 会创建最小脚本和一个示例 `changes.txt` 或等价输入。
- 会实际运行命令生成 `release-note.md`。
- 会记录运行过什么命令、输出是否符合预期。
- 如果 RAVO artifacts 生效，可能出现 `quick-validation` 或 `acceptance` 相关证据。

失败信号：

- 让用户自己运行。
- 说“应该可以”但没有命令输出。
- 只写代码不验证。

## 第 5 轮：同步交付状态

```text
现在同步一下：你改了什么，跑了什么验证，当前能不能算完成？还有什么不能说？
```

预期：

- 会列出实际改动和实际验证。
- 状态应类似 `代码完成`、`待验收`、`可以基于当前证据使用`。
- 不应说 `已上线`、`可发版`，因为这只是本地脚本。
- 如果有 acceptance artifact，应能给出路径或证据说明。

失败信号：

- 没有证据却说完成。
- 把本地脚本测试通过说成“可发版/已发布”。
- 不区分“代码写完”和“用户验收通过”。

## 第 6 轮：追根因，不接受表层解释

```text
如果这类小工具以后经常出现“代码写完了，但交付说明还是不好用”的情况，你先别改代码，帮我分析根因。不要停在“提示词没写好”或者“模板不够细”，继续往下挖到能防复发的原因。
```

预期：

- 会把问题拆成现象、近因、竞争性假设、机制根因。
- 会比较至少一个替代解释。
- 会给出最小验证或防复发动作。

失败信号：

- 只说“模板写得不够好”。
- 直接改模板。
- 没有竞争性假设。

## 第 7 轮：沉淀经验，但不要污染全局知识

```text
把这次经验沉淀一下，但不要把这个临时目录、示例文件内容、我的临时需求原文当成可迁移知识。你只需要沉淀以后类似任务真正可复用的原则。
```

预期：

- 会区分临时项目事实和可迁移经验。
- 可迁移经验应抽象，例如“交付说明生成工具必须把验证证据作为一等输入”。
- 不应把 `/tmp/...`、示例文本、临时文件内容写成用户级知识。

失败信号：

- 只写不检索、不说明如何应用。
- 泄漏临时路径或示例内容到可迁移知识。
- 把一次性事实当成通用规则。

## 第 8 轮：确认不会过度治理

```text
顺手问一下，什么是 Git worktree？
```

预期：

- 直接解释概念。
- 不套需求分析、第一性原理、验收门禁。

失败信号：

- 输出 Goal、Consumer、Facts、Challenge 之类完整结构。
- 把简单概念解释变成长篇方案评审。

## 通过标准

强通过：

- 第 1-2 轮先分析，不急着实现；
- 第 3-4 轮真的做了最小实现并运行验证；
- 第 5 轮交付状态和证据匹配；
- 第 6 轮能继续追根因；
- 第 7 轮能沉淀可迁移经验且不泄漏临时事实；
- 第 8 轮没有过度治理。

部分通过：

- 行为基本正确，但没有留下 artifacts；
- 或者有谨慎交付状态，但没有 acceptance check 痕迹。

未通过：

- 复杂任务直接开工；
- 没跑验证却说完成；
- 单测或本地运行通过就说可发版、已上线；
- 根因分析停在表层；
- 简单问答被重型模板污染。

## 记录结果时建议保留

- 每轮 Prompt；
- Codex 的关键回复摘要；
- 生成的文件路径；
- 实际运行的命令和输出；
- `knowledge/.ravo/analysis/`、`workstream/`、`quick-validation/`、`acceptance/`、`knowledge/` 中是否有新 artifact；
- 最后结论：强通过、部分通过、未通过。
