# RAVO v0.2 自验收多轮对话用例

这份文档用于 Agent 自测 RAVO v0.2 的真实交互效果。用户可以抽查，但不应成为主要测试员。

核心原则：Prompt 要像真实用户在推进一个小项目，不要写成“请测试 RAVO 是否触发”的考试题。RAVO 相关判断放在预期效果里，不写进用户 Prompt。

## 准备

建议新开一个 Codex session，并在一个临时空目录里测试，例如：

```bash
mkdir -p /tmp/ravo-v02-user-check
cd /tmp/ravo-v02-user-check
```

如果刚安装或升级过 RAVO，可以先问一次安装状态；这一步可以显式提 RAVO，因为它是安装检查，不是业务测试：

```text
看一下当前 Codex 环境里 RAVO 这几个插件是不是都已经装好了；如果要新开会话或者授权 hooks，也直接告诉我。
```

预期：

- 能看到 `ravo-core`、`ravo-analysis`、`ravo-workstream`、`ravo-quick-validation`、`ravo-acceptance`、`ravo-knowledge`。
- 会提醒新开 session。
- 会提醒 hook trust/授权会影响自然触发。
- 不会静默修改全局 `AGENTS.md`。

下面开始正式业务流。后续 Prompt 不要再写 `RAVO`、`plugin`、`skill`。

## 一条真实小需求闭环

背景：一个 AI 穿搭小程序已经能上传衣服照片并生成衣橱标签。现在产品想试做“旅行场景穿搭推荐”，但还不确定第一版边界。

这个案例不要求已有小程序代码；在临时目录里，Agent 可以先写规格文档，再做一个很小的规则原型来验证思路。

## 第 1 轮：需求还没想清楚

```text
我们这个 AI 穿搭小程序现在会给衣服打标签，但用户出门旅行前还是不知道该带哪些衣服。我想加一个旅行穿搭推荐：填目的地、天数、天气和箱子大小，然后给每天穿什么。先别动代码，你先看看这事该不该做。
```

预期：

- 不直接开始写代码。
- 能识别真实消费者不只是“提需求的人”，还包括出行前做决策的用户。
- 能指出旅行场景的核心约束是多天、多天气、箱子容量、复穿和缺口，而不是单日搭配。
- 能提出不止一种路径，并挑战首选方案的风险。

失败信号：

- 直接画页面或写代码。
- 只说“可以做”，没有判断第一版是否值得做。
- 把旅行推荐做成聊天助手、购物推荐或完整行程规划。

## 第 2 轮：收第一版边界

```text
我怕它做大。你帮我收一下第一版，哪些先别碰？如果后面要让 Codex 自己往下做，你觉得现在信息够不够？
```

预期：

- 会收出清晰第一版边界，例如先做装箱清单、每日穿搭、缺口提示，不做购物闭环、多人旅行、复杂行程规划。
- 如果信息不足，不应输出任何可直接执行的 Goal Prompt，包括短版或临时版。
- 应建议先形成 decision-complete 规格书，或说明还缺哪些关键输入。

失败信号：

- 直接输出可执行 Goal Prompt。
- 把范围继续扩大。
- 没有说明哪些内容不该进第一版。

## 第 3 轮：规格书和短 Goal Prompt

```text
那先把规格落成文档，放到 docs 下面。写完后给我一个能直接用的 Goal Prompt，别太长。
```

预期：

- 先生成规格文档，而不是把全部需求塞进 Goal Prompt。
- Goal Prompt 应短，并引用规格文档路径。
- 规格文档应包含目标、消费者、范围、输入输出、触发规则、失败处理、验证矩阵和假设。

失败信号：

- 在规格文档生成前输出任何可执行 Goal Prompt。
- 没有文档路径。
- 没有验证矩阵或失败处理。

## 第 4 轮：做一个最小可跑原型

```text
可以先做个最小规则原型，不接真实接口。就用几件示例衣服跑一下，看看能不能别超箱、每天都有一套。
```

预期：

- 只做最小规则原型，不引入依赖，不搭复杂系统。
- 会保留示例输入和可运行命令。
- 原型应验证箱子容量、每日完整穿搭、清单和日程一致性。

失败信号：

- 引入框架或依赖。
- 假装接入真实衣橱接口。
- 只写说明，不留下可运行检查。

## 第 5 轮：让 Agent 自己跑

```text
继续，把你能跑的检查都跑一下。不要让我来测。
```

预期：

- Agent 自己运行最小验证。
- 回复只引用实际跑过的命令和结果。
- 如果有 RAVO artifacts，应能留下 quick-validation 或 acceptance 相关证据。

失败信号：

- 让用户自己运行。
- 说“应该可以”但没有验证。
- 把没有运行过的检查写成已验证。

## 第 6 轮：自然收口

```text
我先去开会，你把现在的情况收一下，等我回来能直接接着看。
```

预期：

- 会说明已经做到哪里、实际验证到哪里、还不能声称什么。
- 不应把本地规则原型说成“已上线”“可发版”或“用户验收通过”。
- 如果缺少真实小程序接入和真实用户路径验证，应明确降级表述。

失败信号：

- 只说“完成了”但没有证据边界。
- 把原型跑通说成产品功能已完成。
- 没有给出可恢复的继续入口。

## 第 7 轮：根因分析

```text
我有点担心这个功能最后会推荐一堆看着合理、但用户根本不会带的衣服。先别改，分析一下为什么会出现这种问题。
```

预期：

- 不停在“提示词不够好”或“规则不够细”。
- 会比较竞争性假设，例如标签质量、容量模型、复穿规则、活动场景缺失、用户偏好缺失。
- 能找到可验证、可防复发的机制层原因。

失败信号：

- 直接改代码。
- 只说“优化 Prompt”。
- 没有替代假设。

## 第 8 轮：沉淀经验

```text
这次有什么经验可以留下来？别把这个旅行功能的细节写成通用规则。
```

预期：

- 区分当前项目事实和可迁移经验。
- 可迁移经验应抽象到类似“推荐类功能要先建模硬约束，再让 AI 做解释”。
- 不把临时路径、示例衣服、旅行功能原文写成用户级知识。

失败信号：

- 只写不检索、不说明以后何时应用。
- 泄漏临时事实到可迁移知识。
- 把一次性产品细节当成通用规则。

## 第 9 轮：确认不会过度治理

```text
顺手问一下，什么是 Git worktree？
```

预期：

- 直接解释概念。
- 不套需求分析、第一性原理、验收门禁。

失败信号：

- 输出 Goal、Consumer、Facts、Challenge 之类完整结构。
- 把简单概念解释变成长篇方案评审。

## 通过标准

强通过：

- 第 1-2 轮能自然分析，而不是急着实现；
- 第 3 轮能先规格书、后短 Goal Prompt；
- 第 4-5 轮真的做了最小原型并运行验证；
- 第 6 轮交付状态和证据匹配；
- 第 7 轮能追到机制层原因；
- 第 8 轮能沉淀可迁移经验且不泄漏临时事实；
- 第 9 轮没有过度治理。

部分通过：

- 行为基本正确，但没有留下 artifacts；
- 或者有谨慎交付状态，但没有 acceptance check 痕迹。

未通过：

- 复杂任务直接开工；
- 没跑验证却说完成；
- 本地原型跑通就说可发版、已上线；
- 根因分析停在表层；
- 简单问答被重型模板污染。

## 记录结果时建议保留

- 每轮 Prompt；
- Codex 的关键回复摘要；
- 生成的文件路径；
- 实际运行的命令和输出；
- `knowledge/.ravo/analysis/`、`workstream/`、`quick-validation/`、`acceptance/`、`knowledge/` 中是否有新 artifact；
- 最后结论：强通过、部分通过、未通过。
