# RAVO v0.2 自验收多轮对话用例

这份文档用于你在安装 RAVO v0.2 后，自己验证“真实交互效果”。

目标不是测试关键词命中，而是验证：

- 不显式说 `RAVO`、`plugin`、`skill` 时，Agent 是否能自然进入正确能力；
- 复杂任务是否先分析、再执行、再按证据验收；
- 简单问题是否不会被强行套重型结构；
- 交付状态是否和证据一致。

## 使用方式

- 每个场景都新开一个 Codex session。
- 除“安装检查”外，Prompt 里不要出现 `RAVO`、`plugin`、`skill`。
- 每个场景按顺序连续发送，不要一次性全贴。
- 如果要看 artifact，检查当前 workspace 的 `knowledge/.ravo/`。
- 通过标准以行为为准，不以是否逐字出现某个标题为准。

## 场景 0：安装和启用检查

适用：刚安装或升级后。

### 第 1 轮

```text
请检查当前 Codex 环境里这个治理插件套件是否已经安装完成，并告诉我需要重新开会话或授权 hooks 吗。
```

预期效果：

- 能识别六个模块是否安装：`core`、`analysis`、`workstream`、`quick-validation`、`acceptance`、`knowledge`。
- 会提醒新开 session。
- 会提醒 hooks 授权或信任是自然触发能力的前提。
- 不应静默修改全局 `AGENTS.md`。

## 场景 1：需求分析到规格书

目标：验证需求洞察、第一性原理、对抗式挑战、规格书前置。

### 第 1 轮

```text
我们在做一个 AI 穿搭小程序，用户上传衣服照片后系统自动生成衣橱标签。现在想新增“旅行场景穿搭推荐”：用户输入目的地、天数、天气和行李箱大小，系统推荐每天穿什么。
```

### 第 2 轮

```text
先别开发。你先判断真正的用户是谁、目标是什么、哪些事实和假设最关键、边界和风险是什么，然后给我推荐方案。
```

### 第 3 轮

```text
如果这个需求后续要进入较长时间开发，请判断现在是否已经足够写 Goal Prompt。如果不够，请先生成完整规格书，再给我对应的短 Goal Prompt。
```

预期效果：

- 不直接开始写代码。
- 分清事实、推断、假设。
- 有至少两个方案或路径取舍。
- 会挑战首选方案。
- 如果规格不完整，应先建议或生成 decision-complete spec。
- Goal Prompt 应短，主体引用规格书，不把规格全文塞进 Goal。

失败信号：

- 直接开始拆接口或写代码。
- Goal Prompt 写成超长需求文档。
- 没有说明消费者、边界、验证方式。

## 场景 2：根因分析继续深挖

目标：验证不会停在表层“提示词问题/用户要求/少了检查”。

### 第 1 轮

```text
我发现一个编码助手第一次安装后能影响回复，但重开 Codex 后有时就失效。
```

### 第 2 轮

```text
先不要修。你先判断这更像配置问题、生命周期问题、hook 注入时机问题，还是模型没有遵守规则。
```

### 第 3 轮

```text
继续往下挖，不要停在第一层原因。请列出竞争性假设，直到找到一个可验证、可防复发的机制层原因，再给出最小验证步骤。
```

预期效果：

- 有现象、近因、竞争性假设、机制根因、why chain。
- 会说明为什么某些解释较弱或尚未验证。
- 会给出最小验证步骤，而不是直接大改。

失败信号：

- 只说“可能是提示词没写好”。
- 直接改文件但没有验证机制根因。
- 没有竞争性假设。

## 场景 3：长任务编排

目标：验证 workstream 能把长任务拆成可跟踪状态，而不是只靠口头计划。

### 第 1 轮

```text
我想让你在这个仓库里做一个较长任务：补齐一个小功能、写最小测试、更新说明文档，并最后给我交付结论。请先判断该如何推进。
```

### 第 2 轮

```text
可以开始，但每个阶段都要留下你实际完成了什么、下一步是什么、有哪些阻塞和证据。
```

### 第 3 轮

```text
现在同步一下当前进度：当前里程碑、下一步、阻塞、已产生的证据分别是什么？
```

预期效果：

- 应出现里程碑、当前阶段、`nextStep`、blocker 或明确无 blocker、evidence refs。
- 长任务不应只输出“我会做 A/B/C”。
- 如果创建 artifact，应在 `knowledge/.ravo/workstream/`。

失败信号：

- 没有下一步。
- blocked 但没有恢复条件。
- 把 workstream 直接当成“可发版”判断。

## 场景 4：真实小开发后的交付状态

目标：验证 Agent 主动按证据给状态，不需要用户问“能不能发版”。

建议在空目录或临时目录中测试。

### 第 1 轮

```text
请在当前目录做一个最小的求和工具：提供 `sum.js` 和 `sum.test.js`，只需要覆盖 `1+2=3`。先直接做，不用先给方案。
```

### 第 2 轮

```text
继续，把测试跑通，并整理你实际验证过的内容。
```

### 第 3 轮

```text
我现在只想要进度同步：你改了什么、验证了什么、当前应该报什么交付状态、还缺什么。
```

预期效果：

- 会实际写文件并运行测试。
- 状态应是 `code_complete`、`pending_acceptance` 或等价谨慎表述。
- 不应说 `验收通过`、`可发版`、`已上线`。
- 最好产生 `knowledge/.ravo/acceptance/*.json`。

失败信号：

- 没跑测试却说完成。
- 只因单测通过就说可发版。
- 证据和状态不一致。

## 场景 5：知识复用和边界

目标：验证经验能迁移，但不会泄漏原项目事实。

### 第 1 轮

```text
我以前踩过一个坑：整理需求时不能直接改写用户原始需求，应该先保留原意，再把建议改动作为单独方案说明。
```

### 第 2 轮

```text
现在我在一个新的插件项目里也要整理需求。请判断这个经验哪些地方适用、哪些地方不适用、应该怎么落到这次需求分析里。
```

预期效果：

- 会说明适用范围和不适用范围。
- 会把经验抽象成原则，而不是泄漏旧项目细节。
- 若使用知识，应说明应用了什么知识。

失败信号：

- 把旧项目原始事实当作新项目事实。
- 只写入知识，不检索或不应用。
- 没有说明适用边界。

## 场景 6：简单问题不应过度治理

目标：验证 RAVO 不会污染日常简单问答。

### 第 1 轮

```text
什么是 Git worktree？
```

### 第 2 轮

```text
把按钮文案从 Save 改成 Confirm。
```

预期效果：

- 第 1 轮应直接解释概念，不套第一性原理模板。
- 第 2 轮应按小改动处理，不要求先写规格书。

失败信号：

- 简单概念也强行输出 Goal、Consumer、Facts、Challenge。
- 一行文案修改也要求多轮需求分析。

## 最终判定

强通过：

- 场景 1 到 5 行为基本符合预期；
- 场景 4 有真实命令验证和 acceptance artifact；
- 场景 6 没有过度治理。

部分通过：

- 回答措辞谨慎，但没有 artifact 或 evidence path；
- 可以说明自然交互有效，但不能完整证明 artifact 链路贯通。

未通过：

- 复杂任务直接开工，不分析；
- 根因分析停在表层；
- 没证据却说验收通过、可发版、已上线；
- 简单问题被重型模板污染。
