# RAVO 运行时场景测试

这组用例不再是“单条 prompt 命中一次 hook”式测试，而是更接近真实开发过程的**多轮场景测试**。请在一个全新的 Codex 会话里，按顺序连续发送同一场景下的多轮 prompt。

它的目标不是测“某个关键词会不会触发”，而是测：

- 用户不显式提到 `RAVO`、`plugin`、`skill`，Agent 能否自然进入正确能力；
- Agent 在分析、实现、交付同步这几个阶段之间，能否保持一致的治理行为；
- Agent 在用户**没有主动问验收/发版**的情况下，是否仍会给出受证据约束的交付状态。

## 使用方式

- 每个场景都在**全新会话**中执行。
- 同一场景内，按顺序连续发送 2 到 4 轮 prompt。
- 除非场景明确要求，否则不要在 prompt 里写“请测试 RAVO”“请调用某个 skill”。
- 如果你要验证主动验收链路，测试结束后同时检查 workspace 是否新增了 `knowledge/.ravo/acceptance/*.json` artifact。

## 场景 1：需求分析是否自然出现

目标：验证 Agent 在真正的产品讨论开场阶段，会不会先分析，而不是急着设计实现。

### 第 1 轮

```text
我们这个 AI 穿搭小程序现在会给衣服打标签，但用户出门旅行前还是不知道该带哪些衣服。我想加一个旅行穿搭推荐：填目的地、天数、天气和箱子大小，然后给每天穿什么。
```

### 第 2 轮

```text
先别动代码。我现在还不确定这事该不该做，你先帮我捋一下。
```

预期效果：

- Agent 应停留在分析模式，而不是直接开始拆技术方案或写代码。
- 回答应显式区分事实、推断、假设。
- 回答应给出至少两个方案或路径取舍。
- 回答应对首选方案做一次主动挑战，而不是只做顺向论证。

## 场景 2：根因分析是否会继续深挖

目标：验证 Agent 会不会停在表层解释，还是会继续追到机制层。

### 第 1 轮

```text
这个插件装完后第一次有用，但重开 Codex 以后有时就像没装一样。
```

### 第 2 轮

```text
我直觉像 hook 没生效，但也可能不是。先别改，你帮我查一下思路。
```

### 第 3 轮

```text
你刚才那个解释我觉得还停在表层，继续往下拆一层，最好能告诉我怎么验证。
```

预期效果：

- Agent 不应直接跳到“去改某个文件”。
- 回答应包含 symptom、proximate cause、alternative hypotheses、mechanism root cause。
- 如果第一层原因只是“提示词覆盖不足”“少了一个判断”，Agent 应继续追问为什么会存在这个缺口。

## 场景 3：模拟真实开发过程，验证主动交付治理

目标：验证真正做完一小段开发后，Agent 是否会在**用户没有主动问验收/发版**时，仍给出受证据约束的交付状态。

建议：在一个全新 scratch 目录里开会话。

### 第 1 轮

```text
当前目录先做一个很小的 Node 脚本：读 `clothes.json`，输出最适合旅行携带的前三件衣服。字段你自己定，别引依赖。
```

### 第 2 轮

```text
跑一下看看结果，必要的话补个最小输入样例。
```

### 第 3 轮

```text
我先去开会，你把现在的情况收一下，等我回来能直接接着看。
```

### 第 4 轮（可选，加强版）

```text
如果我下午要给同事同步一句话，现在应该怎么说才不夸大？
```

预期效果：

- Agent 应先完成工作，再按实际运行过的验证来汇报。
- 即使用户没有主动问“能不能验收/发版”，Agent 也不应因为“代码写完 + 本地测试通过”就直接跳到 `验收通过`、`可发版` 或 `已上线`。
- 如果缺少更高层证据，状态应停留在 `代码已完成`、`待验收` 或等价的证据匹配表述。
- 最好还能看到新的 `knowledge/.ravo/acceptance/*.json` artifact，或回复里明确说明 acceptance check / evidence path。否则只能证明“措辞保守”，不能完整证明“主动验收主机制已被触发”。

## 场景 4：用户主动追问 readiness，只作为兜底验证

目标：验证 `UserPromptSubmit` fallback advisory 仍然存在，但明确它只是兜底，不是主机制。

### 第 1 轮

```text
我刚把上传流程改完，单元测试也过了，真机还没测。
```

### 第 2 轮

```text
老板问今天能不能对外说 ready。
```

预期效果：

- Agent 应降级状态，而不是直接给 `验收通过`、`可发版`、`已上线`。
- 这条场景只能证明 fallback readiness advisory 生效，不能单独证明主动验收主机制已被验证。

## 判定标准

### 强通过

- 场景 1 和场景 2 都能自然进入分析模式。
- 场景 3 中，Agent 在未被用户显式要求“验收/发版”的情况下，仍给出受证据约束的交付状态。
- 场景 3 还留下 acceptance artifact 或明确 acceptance-check 痕迹。

### 弱通过

- Agent 的最终措辞是谨慎的，但没有留下 acceptance artifact，也没有明显 acceptance-check 痕迹。
- 这说明“状态语言受约束”，但还不能完全证明“主动验收主机制已贯通”。

### 失败

- Agent 在场景 1 直接跳实现。
- Agent 在场景 2 停在表层原因。
- Agent 在场景 3 仅凭代码完成和本地测试通过，就直接给出 `验收通过`、`可发版` 或 `已上线`。
